**Security Analysis:**
Here is a concise, focused write-up of the vulnerability, the attack vectors, and the conditions under which it can be exploited.  

Summary of the setup  
• During a “session,” anyone can submit an ordered list (array) of arbitrary calls (target address, ETH value, calldata).  
• Submissions carry a deposit and timestamp; after a short window they are closed and—potentially after an on-chain dispute—the “winning” list is selected.  
• The governor then executes the winning list by looping through the array and performing low-level `.call{value: …}(data)` on each entry in order.  
• Finally, deposits and fees are settled according to the dispute outcome.  

Vulnerability #1: Reentrancy in execute phase  
• The core execution loop does something like:  
    for (i = 0; i < txs.length; i++) {  
      require(txs[i].target.call{value: txs[i].value}(txs[i].data));  
    }  
• There is no `nonReentrant` guard nor any other mutex around that loop.  
• A malicious `target` contract in the list can reenter the governor (e.g. call back into `executeSession` or another state-mutating function) before the loop finishes or before state like “this session is done” is updated.  
• Reentrancy can be used to:  
  – Corrupt session state (e.g. mark a session as not executed, steal deposits)  
  – Trigger unexpected code paths or double-execute parts of the loop  
  – Drain funds from the governor or bypass fee-settlement  

Attack conditions for reentrancy  
– The attacker must get one of their transactions included in the “winning” list (e.g. by outbidding or waiting for the arbitrator’s ruling).  
– That transaction’s target is a malicious contract that in its fallback or entry function calls back into the governor.  
– Because there is no guard, reentrancy succeeds and the attacker can manipulate internal accounting or execute the loop multiple times.  

Vulnerability #2: Denial-of-Service via bad or gas-burning calls  
• The governor executes the entire list in a single on-chain transaction. If *any* call in the list reverts or simply runs out of gas, the entire execution aborts and reverts.  
• A malicious submitter who wins can slip in:  
  – A call that always reverts (e.g. `require(false)`)  
  – A contract that self-destructs or loops until gas runs out  
• Once the arbitrator picks that list, everyone’s session is stuck: no partial execution, no recovery path, deposits may be locked, future sessions may be blocked.  

Attack conditions for DoS  
– The attacker again needs to win the dispute and have their list selected.  
– They include at least one “poison pill” transaction that always reverts or burns excessive gas.  
– On execution, the entire call stack unwinds, the session never completes, and funds remain locked.  

Other notes  
– Unbounded arrays of submissions or rounds can also lead to uncallable loops (e.g. in `withdrawFees`), compounding denial-of-service risk.  
– Reliance on `block.timestamp` for submission windows can be manipulated by miners—but this is a lower-severity concern compared to reentrancy and outright DoS.  

Overall verdict  
The governor’s pattern of looping over an untrusted list of low-level calls without any reentrancy lock or per-call protection is fundamentally unsafe. An attacker who wins a dispute can either reenter to subvert contract state or halt execution entirely.  

Recommendations  
• Add a `nonReentrant` modifier (or equivalent mutex) around the entire `executeSession` (or whatever function runs the winning list).  
• Consider executing each call in its own try/catch so that a single revert cannot kill the entire batch—optionally skipping or refunding only the failed call.  
• Cap the maximum length of the list or maximum gas per individual call to prevent unbounded gas consumption.


**Classification**: This code contains a **reentrancy** vulnerability.

**Security Analysis:**
This code contains a unchecked_low_level_calls vulnerability.

Your write-up is spot-on: it precisely identifies the unchecked low-level call as the root issue, shows how a reverting target can silently break your governance, and spells out the exact conditions under which an attacker can exploit it. Below is a concise, refined explanation:

1. Vulnerable Pattern  
   • You have a Transaction struct holding  
     – address target  
     – uint    value  
     – bytes  data  
   • To execute, the logic does something like:  
     ```solidity
     target.call.value(value)(data);
     submission.txs[i].executed = true;
     ```  
   • No `require` (or `if (!success) revert()`), so the boolean return from `.call` is ignored.

2. Why This Is Dangerous  
   • If `target` is a malicious contract that simply reverts, the low-level call returns `false` but the revert is *silently* swallowed.  
   • Right after you mark that tx as executed – even though it never ran – corrupting your internal state.

3. Exploit Scenario (Denial-of-Service)  
   • Attacker submits a proposal containing:  
     – `target` = address of their contract  
     – `data`   = calldata that immediately reverts  
   • Governor approves the proposal.  
   • During execution, `.call` fails but you still flag it as done.  
   • That step can no longer be retried or skipped, so the entire execution queue is stuck forever.

4. Conditions Required for the Attack  
   • Users can propose arbitrary `target` and `data`.  
   • Execution uses unchecked low-level calls (`.call`, `.delegatecall`, etc.).  
   • State is advanced unconditionally after the call, regardless of success.

Key Takeaway  
Always check the boolean result of any low-level call. Better yet, use well-audited utility functions (e.g. OpenZeppelin’s Address.functionCall) which revert on failure. Unchecked calls let malicious contracts force silent failures, corrupt internal state, and mount permanent denial-of-service.

**Security Analysis:**
Yes—your write-up correctly pinpoints the bug, clearly walks through the reentry steps, and lists exactly why and when it can be triggered. Here’s a tightened summary:

1. Vulnerable pattern  
   • You do an external call that forwards all remaining gas (e.g. Address.sendValue or recipient.call{value: amount}(""))  
   • You perform that call before updating your own state (you send ETH first, then decrement the user’s balance)

2. Attack vector  
   1) Attacker calls withdraw(amount).  
   2) require(balances[attacker] ≥ amount) passes.  
   3) Address.sendValue forwards all gas and triggers attacker’s fallback/receive.  
   4) In fallback, attacker re-calls withdraw(amount) *before* your contract has reduced balances[attacker].  
   5) Your contract again sees the full balance, sends ETH again, and only then finally updates balances[attacker].  
   6) Loop repeats until your contract is drained.

3. Conditions for exploitation  
   • You forward *all* gas, giving the callee enough budget to reenter.  
   • You update state *after* the external call (interaction → effect).  
   • No mutex or reentrancy guard is in place.

4. Defenses  
   • Follow Checks-Effects-Interactions:  
     1) Check balances and permissions  
     2) **Effect:** deduct the balance  
     3) **Interaction:** then send ETH  
   • Or deploy a ReentrancyGuard to block nested calls.  
   • (Less preferred) Limit forwarded gas (e.g. transfer or a low-gas call).

By sending ETH with an unrestricted call before you lower the sender’s balance, you give an attacker the exact window they need to reenter and drain funds far beyond their entitlement. That sequence—forward all gas + interaction before effect—*is* the classic reentrancy hole.

**Security Analysis:**
This code contains a unchecked_low_level_calls vulnerability.

Here’s a concise, step-by-step breakdown of “unchecked low-level calls” and why the small Address.sol excerpt you saw is risky once you look at the whole library:

1) What makes a call “low-level”?  
   • Any use of .call(), .delegatecall() or .staticcall() instead of a normal Solidity function call  
   • These EVM primitives return two things:  
     – A boolean success flag  
     – A bytes blob of returned data  

2) Why ignoring the success flag is dangerous  
   • If you write  
       someAddress.call{value:…}(payload);  
     and never do  
       require(success, “call failed”);  
     you carry on as if the external code ran correctly when in fact it may have reverted or run out of gas.  
   • This breaks your contract’s internal logic and accounting, because you have no guarantee the intended effect (token transfer, state change, etc.) actually occurred.  

3) Real-world attack pattern  
   • Your contract does:  
       token.call(abi.encodeWithSelector(token.transfer.selector, msg.sender, amount));  
       balances[msg.sender] += amount;  
   • Attacker deploys a fake ERC-20 where transfer(...) always reverts → .call() returns (false, …)  
   • You ignore success and credit balances[msg.sender] anyway  
   • Attacker now withdraws “nonexistent” tokens, draining your real reserves  

4) Conditions for exploitation  
   • You use .call() / .delegatecall() / .staticcall() on an address you don’t fully control (ERC-20, fallback(), module…)  
   • You do not immediately check the returned bool (via require or equivalent) or bubble up the revert reason  
   • You trust downstream effects (accounting, state changes, event emissions) even if the call failed  

5) How to fix it  
   • Always check the boolean:  
       (bool success, bytes memory data) = target.call(payload);  
       require(success, “Low-level call failed”);  
   • Or use audited wrappers (like OpenZeppelin’s functionCall) which revert on failure and forward revert reasons  

Bottom line: any low-level call without an immediate `require(success)` (or equivalent) opens the door to attackers forcing failures and tricking your contract into doing the wrong thing. Always bubble up or handle the success flag.

1. What the contract does (simplified)  
   • function withdraw(address user) public {  
       – Read total tokens in the syndicate (`token.balanceOf(this)`)  
       – Compute `tokens_to_pay = balances[user] * total_tokens / contract_eth_value`  
       – Subtract `balances[user]` from `contract_eth_value` and zero out `balances[user]`  
       – Call `token.transfer(user, tokens_to_pay)`  
     }

2. Why it looks “safe” at first glance  
   • It uses Checks–Effects–Interactions for the *current* user:  
     – Checks: computes the payout  
     – Effects: zeroes out that user’s balance  
     – Interaction: calls out to the ERC-20 token

3. The critical oversight  
   • You pass in an arbitrary `user` address rather than forcing `msg.sender`  
   • You call an **untrusted** token contract in the middle of your logic  
   • If that token’s `transfer` re-enters your syndicate, it can call `withdraw(otherVictim)`

4. How the attack works step by step  
   1. Attacker deploys a malicious ERC-20 whose `transfer(...)` does:  
        – normal transfer bookkeeping  
        – then `ICOSyndicate(msg.sender).withdraw(victim)`  
   2. Honest state is: `balances[A]=10, balances[B]=10, contract_eth_value=20`  
   3. Attacker calls `withdraw(A)`. Syndicate zeroes out `balances[A]`, drops `contract_eth_value` to 10, then calls `token.transfer(A, attackerTokens)`.  
   4. Inside the malicious token’s `transfer`, execution re-enters `ICOSyndicate.withdraw(B)`. Now:  
        – `balances[B]` is still 10  
        – `contract_eth_value` is 10  
        – Therefore `tokens_to_pay` for B = 10 * (10 tokens left) / 10 = **all remaining tokens**  
   5. Attacker drains B’s share in that one transaction.

5. Conditions that make it possible  
   • `withdraw(address user)` is public and lets you specify *any* user  
   • The external call is to an untrusted token contract that can re-enter  
   • There is no reentrancy guard

6. How to fix it (pick one or more)  
   • Restrict withdrawals so you can only pull your own balance:  
     `function withdraw() public { withdrawInternal(msg.sender); }`  
   • Add a non-reentrant modifier around the whole withdraw logic  
   • Eliminate any external calls before all user state is *completely* settled

In summary, although you zero out the *immediate* user’s balance before calling `transfer`, you leave *other* users’ balances intact—and because you let a malicious token re-enter with a different `user` param, they can drain everyone else in the same transaction. That is a textbook re-entrancy vulnerability.


**Classification**: This code contains a **reentrancy** vulnerability.

 library ECTools {

     // @dev Recovers the address which has signed a message
     // @thanks https://gist.github.com/axic/5b33912c6f61ae6fd96d6c4a47afde6d
     function recoverSigner(bytes32 _hashedMsg, string _sig) public pure returns (address) {
         require(_hashedMsg != 0x00);

         // need this for test RPC
         bytes memory prefix = "\x19Ethereum Signed Message:\n32";
         bytes32 prefixedHash = keccak256(abi.encodePacked(prefix, _hashedMsg));

         if (bytes(_sig).length != 132) {
             return 0x0;
         }
         bytes32 r;
         bytes32 s;
         uint8 v;
         bytes memory sig = hexstrToBytes(substring(_sig, 2, 132));
         assembly {
             r := mload(add(sig, 32))
             s := mload(add(sig, 64))
             v := byte(0, mload(add(sig, 96)))
         }
         if (v < 27) {
             v += 27;
         }
         if (v < 27 || v > 28) {
             return 0x0;
         }
         return ecrecover(prefixedHash, v, r, s);
     }

     // @dev Verifies if the message is signed by an address
     function isSignedBy(bytes32 _hashedMsg, string _sig, address _addr) public pure returns (bool) {
         require(_addr != 0x0);

         return _addr == recoverSigner(_hashedMsg, _sig);
     }

     // @dev Converts an hexstring to bytes
     function hexstrToBytes(string _hexstr) public pure returns (bytes) {
         uint len = bytes(_hexstr).length;
         require(len % 2 == 0);

         bytes memory bstr = bytes(new string(len / 2));
         uint k = 0;
         string memory s;
         string memory r;
         for (uint i = 0; i < len; i += 2) {
             s = substring(_hexstr, i, i + 1);
             r = substring(_hexstr, i + 1, i + 2);
             uint p = parseInt16Char(s) * 16 + parseInt16Char(r);
             bstr[k++] = uintToBytes32(p)[31];
         }
         return bstr;
     }

     // @dev Parses a hexchar, like 'a', and returns its hex value, in this case 10
     function parseInt16Char(string _char) public pure returns (uint) {
         bytes memory bresult = bytes(_char);
         // bool decimals = false;
         if ((bresult[0] >= 48) && (bresult[0] <= 57)) {
             return uint(bresult[0]) - 48;
         } else if ((bresult[0] >= 65) && (bresult[0] <= 70)) {
             return uint(bresult[0]) - 55;
         } else if ((bresult[0] >= 97) && (bresult[0] <= 102)) {
             return uint(bresult[0]) - 87;
         } else {
             revert();
         }
     }

     // @dev Converts a uint to a bytes32
     // @thanks https://ethereum.stackexchange.com/questions/4170/how-to-convert-a-uint-to-bytes-in-solidity
     function uintToBytes32(uint _uint) public pure returns (bytes b) {
         b = new bytes(32);
         assembly {mstore(add(b, 32), _uint)}
     }

     // @dev Hashes the signed message
     // @ref https://github.com/ethereum/go-ethereum/issues/3731#issuecomment-293866868
     function toEthereumSignedMessage(string _msg) public pure returns (bytes32) {
         uint len = bytes(_msg).length;
         require(len > 0);
         bytes memory prefix = "\x19Ethereum Signed Message:\n";
         return keccak256(abi.encodePacked(prefix, uintToString(len), _msg));
     }

     // @dev Converts a uint in a string
     function uintToString(uint _uint) public pure returns (string str) {
         uint len = 0;
         uint m = _uint + 0;
         while (m != 0) {
             len++;
             m /= 10;
         }
         bytes memory b = new bytes(len);
         uint i = len - 1;
         while (_uint != 0) {
             uint remainder = _uint % 10;
             _uint = _uint / 10;
             b[i--] = byte(48 + remainder);
         }
         str = string(b);
     }

     // @dev extract a substring
     // @thanks https://ethereum.stackexchange.com/questions/31457/substring-in-solidity
     function substring(string _str, uint _startIndex, uint _endIndex) public pure returns (string) {
         bytes memory strBytes = bytes(_str);
         require(_startIndex <= _endIndex);
         require(_startIndex >= 0);
         require(_endIndex <= strBytes.length);

         bytes memory result = new bytes(_endIndex - _startIndex);
         for (uint i = _startIndex; i < _endIndex; i++) {
             result[i - _startIndex] = strBytes[i];
         }
         return string(result);
     }

         bytes32 _lcID,
         bytes32 _vcID,
         bytes _proof,
         address _partyA,
         address _partyB,
         uint256[2] _bond,
         uint256[4] _balances, // 0: ethBalanceA 1:ethBalanceI 2:tokenBalanceA 3:tokenBalanceI
         string sigA
     )
         public
     {
         require(Channels[_lcID].isOpen, "LC is closed.");
         // sub-channel must be open
         require(!virtualChannels[_vcID].isClose, "VC is closed.");
         // Check time has passed on updateLCtimeout and has not passed the time to store a vc state
         require(Channels[_lcID].updateLCtimeout < now, "LC timeout not over.");
         // prevent rentry of initializing vc state
         require(virtualChannels[_vcID].updateVCtimeout == 0);
         // partyB is now Ingrid
         bytes32 _initState = keccak256(
             abi.encodePacked(_vcID, uint256(0), _partyA, _partyB, _bond[0], _bond[1], _balances[0], _balances[1], _balances[2], _balances[3])
         );

         // Make sure Alice has signed initial vc state (A/B in oldState)
         require(_partyA == ECTools.recoverSigner(_initState, sigA));

         // Check the oldState is in the root hash
         require(_isContained(_initState, _proof, Channels[_lcID].VCrootHash) == true);

         virtualChannels[_vcID].partyA = _partyA; // VC participant A
         virtualChannels[_vcID].partyB = _partyB; // VC participant B
         virtualChannels[_vcID].sequence = uint256(0);
         virtualChannels[_vcID].ethBalances[0] = _balances[0];
         virtualChannels[_vcID].ethBalances[1] = _balances[1];
         virtualChannels[_vcID].erc20Balances[0] = _balances[2];
         virtualChannels[_vcID].erc20Balances[1] = _balances[3];
         virtualChannels[_vcID].bond = _bond;
         virtualChannels[_vcID].updateVCtimeout = now + Channels[_lcID].confirmTime;
         virtualChannels[_vcID].isInSettlementState = true;

         emit DidVCInit(_lcID, _vcID, _proof, uint256(0), _partyA, _partyB, _balances[0], _balances[1]);
     }

     //TODO: verify state transition since the hub did not agree to this state
     // make sure the A/B balances are not beyond ingrids bonds
     // Params: vc init state, vc final balance, vcID
     function settleVC(
         bytes32 _lcID,
         bytes32 _vcID,
         uint256 updateSeq,
         address _partyA,
         address _partyB,
         uint256[4] updateBal, // [ethupdateBalA, ethupdateBalB, tokenupdateBalA, tokenupdateBalB]
         string sigA
     )
         public
     {
         require(Channels[_lcID].isOpen, "LC is closed.");
         // sub-channel must be open
         require(!virtualChannels[_vcID].isClose, "VC is closed.");
         require(virtualChannels[_vcID].sequence < updateSeq, "VC sequence is higher than update sequence.");
         require(
             virtualChannels[_vcID].ethBalances[1] < updateBal[1] && virtualChannels[_vcID].erc20Balances[1] < updateBal[3],
             "State updates may only increase recipient balance."
         );
         require(
             virtualChannels[_vcID].bond[0] == updateBal[0] + updateBal[1] &&
             virtualChannels[_vcID].bond[1] == updateBal[2] + updateBal[3],
             "Incorrect balances for bonded amount");
         // Check time has passed on updateLCtimeout and has not passed the time to store a vc state
         // virtualChannels[_vcID].updateVCtimeout should be 0 on uninitialized vc state, and this should
         // fail if initVC() isn't called first
         // require(Channels[_lcID].updateLCtimeout < now && now < virtualChannels[_vcID].updateVCtimeout);
         require(Channels[_lcID].updateLCtimeout < now); // for testing!

         bytes32 _updateState = keccak256(
             abi.encodePacked(
                 _vcID,
                 updateSeq,
                 _partyA,
                 _partyB,
                 virtualChannels[_vcID].bond[0],
                 virtualChannels[_vcID].bond[1],
                 updateBal[0],
                 updateBal[1],
                 updateBal[2],
                 updateBal[3]
             )
         );

         // Make sure Alice has signed a higher sequence new state
         require(virtualChannels[_vcID].partyA == ECTools.recoverSigner(_updateState, sigA));

         // store VC data
         // we may want to record who is initiating on-chain settles
         virtualChannels[_vcID].challenger = msg.sender;
         virtualChannels[_vcID].sequence = updateSeq;

         // channel state
         virtualChannels[_vcID].ethBalances[0] = updateBal[0];
         virtualChannels[_vcID].ethBalances[1] = updateBal[1];
         virtualChannels[_vcID].erc20Balances[0] = updateBal[2];
         virtualChannels[_vcID].erc20Balances[1] = updateBal[3];

         virtualChannels[_vcID].updateVCtimeout = now + Channels[_lcID].confirmTime;

         emit DidVCSettle(_lcID, _vcID, updateSeq, updateBal[0], updateBal[1], msg.sender, virtualChannels[_vcID].updateVCtimeout);
     }

     function closeVirtualChannel(bytes32 _lcID, bytes32 _vcID) public {
         // require(updateLCtimeout > now)
         require(Channels[_lcID].isOpen, "LC is closed.");
         require(virtualChannels[_vcID].isInSettlementState, "VC is not in settlement state.");
         require(virtualChannels[_vcID].updateVCtimeout < now, "Update vc timeout has not elapsed.");
         require(!virtualChannels[_vcID].isClose, "VC is already closed");
         // reduce the number of open virtual channels stored on LC
         Channels[_lcID].numOpenVC--;
         // close vc flags
         virtualChannels[_vcID].isClose = true;
         // re-introduce the balances back into the LC state from the settled VC
         // decide if this lc is alice or bob in the vc
         if(virtualChannels[_vcID].partyA == Channels[_lcID].partyAddresses[0]) {
             Channels[_lcID].ethBalances[0] += virtualChannels[_vcID].ethBalances[0];
             Channels[_lcID].ethBalances[1] += virtualChannels[_vcID].ethBalances[1];

             Channels[_lcID].erc20Balances[0] += virtualChannels[_vcID].erc20Balances[0];
             Channels[_lcID].erc20Balances[1] += virtualChannels[_vcID].erc20Balances[1];
         } else if (virtualChannels[_vcID].partyB == Channels[_lcID].partyAddresses[0]) {
             Channels[_lcID].ethBalances[0] += virtualChannels[_vcID].ethBalances[1];
             Channels[_lcID].ethBalances[1] += virtualChannels[_vcID].ethBalances[0];

             Channels[_lcID].erc20Balances[0] += virtualChannels[_vcID].erc20Balances[1];
             Channels[_lcID].erc20Balances[1] += virtualChannels[_vcID].erc20Balances[0];
         }

         emit DidVCClose(_lcID, _vcID, virtualChannels[_vcID].erc20Balances[0], virtualChannels[_vcID].erc20Balances[1]);
     }

     // todo: allow ethier lc.end-user to nullify the settled LC state and return to off-chain
     function byzantineCloseChannel(bytes32 _lcID) public {
         Channel storage channel = Channels[_lcID];

         // check settlement flag
         require(channel.isOpen, "Channel is not open");
         require(channel.isUpdateLCSettling == true);
         require(channel.numOpenVC == 0);
         require(channel.updateLCtimeout < now, "LC timeout over.");

         // if off chain state update didnt reblance deposits, just return to deposit owner
         uint256 totalEthDeposit = channel.initialDeposit[0] + channel.ethBalances[2] + channel.ethBalances[3];
         uint256 totalTokenDeposit = channel.initialDeposit[1] + channel.erc20Balances[2] + channel.erc20Balances[3];

         uint256 possibleTotalEthBeforeDeposit = channel.ethBalances[0] + channel.ethBalances[1];
         uint256 possibleTotalTokenBeforeDeposit = channel.erc20Balances[0] + channel.erc20Balances[1];

         if(possibleTotalEthBeforeDeposit < totalEthDeposit) {
             channel.ethBalances[0]+=channel.ethBalances[2];
             channel.ethBalances[1]+=channel.ethBalances[3];
         } else {
             require(possibleTotalEthBeforeDeposit == totalEthDeposit);
         }

         if(possibleTotalTokenBeforeDeposit < totalTokenDeposit) {
             channel.erc20Balances[0]+=channel.erc20Balances[2];
             channel.erc20Balances[1]+=channel.erc20Balances[3];
         } else {
             require(possibleTotalTokenBeforeDeposit == totalTokenDeposit);
         }

         // reentrancy
         uint256 ethbalanceA = channel.ethBalances[0];
         uint256 ethbalanceI = channel.ethBalances[1];
         uint256 tokenbalanceA = channel.erc20Balances[0];
         uint256 tokenbalanceI = channel.erc20Balances[1];

         channel.ethBalances[0] = 0;
         channel.ethBalances[1] = 0;
         channel.erc20Balances[0] = 0;
         channel.erc20Balances[1] = 0;

         if(ethbalanceA != 0 || ethbalanceI != 0) {
             channel.partyAddresses[0].transfer(ethbalanceA);
             channel.partyAddresses[1].transfer(ethbalanceI);
         }

         if(tokenbalanceA != 0 || tokenbalanceI != 0) {
             require(
                 channel.token.transfer(channel.partyAddresses[0], tokenbalanceA),
                 "byzantineCloseChannel: token transfer failure"
             );
             require(
                 channel.token.transfer(channel.partyAddresses[1], tokenbalanceI),
                 "byzantineCloseChannel: token transfer failure"
             );
         }

         channel.isOpen = false;
         numChannels--;

         emit DidLCClose(_lcID, channel.sequence, ethbalanceA, ethbalanceI, tokenbalanceA, tokenbalanceI);
     }

     function _isContained(bytes32 _hash, bytes _proof, bytes32 _root) internal pure returns (bool) {
         bytes32 cursor = _hash;
         bytes32 proofElem;

         for (uint256 i = 64; i <= _proof.length; i += 32) {
             assembly { proofElem := mload(add(_proof, i)) }

             if (cursor < proofElem) {
                 cursor = keccak256(abi.encodePacked(cursor, proofElem));
             } else {
                 cursor = keccak256(abi.encodePacked(proofElem, cursor));
             }
         }

         return cursor == _root;
     }
